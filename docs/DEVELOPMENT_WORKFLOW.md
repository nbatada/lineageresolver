# Development Workflow

## Working Style
Use an implement-test-correct loop on small increments mapped to ticket scope. Each ticket should end with passing tests and a clear output contract before the next ticket begins.

## Implement-Test-Correct Loop
1. Define scope and acceptance criteria from the active ticket.
2. Add or update tests first for expected behavior where feasible.
3. Implement minimal code path to satisfy tests.
4. Run unit tests and static checks locally.
5. Correct edge cases and diagnostics.
6. Update docs/examples if behavior changed.
7. Merge only when the acceptance criteria are met.

## Branch and PR Conventions
- Branch naming: `codex/<ticket-id>-<short-slug>`.
- Keep each branch focused on one ticket.
- PR description must include:
  - ticket reference
  - summary of behavior changes
  - tests run
  - known limitations or deferred follow-up

## Testing Strategy

### Synthetic Fixture Policy (CI-required)
- All committed test fixtures must live under `tests/data/`.
- Total committed fixture size must remain CI-friendly (target: under 10 MB).
- Do not commit large real 10x datasets; use deterministic synthetic fixtures generated by `tests/make_test_data.py`.
- Fixture generation must use a fixed seed so outputs are reproducible across local runs and CI.

### Fixture Usage
- Raw 10x ambient tests must read `tests/data/raw10x_small/raw_feature_bc_matrix/` and validate ambient estimation behavior on mixed empty/cell barcodes.
- End-to-end infer tests must use:
  - `tests/data/filtered_small.h5ad`
  - `tests/data/evidence_small.tsv`
  - `tests/data/configs/cytotoxic_adjudication_v1.yaml`
- Fixture metadata for barcode partitioning, truth labels, contamination regime, and missing evidence rows is in `tests/data/raw10x_small/manifest.json`.

### Unit Tests (required)
- Ambient estimation correctness on synthetic sparse raw10x-like matrices.
- Fallback ambient mode behavior and warning/diagnostic flags.
- Module scoring correctness with weighted gene sets.
- Poisson-derived `amb_ll_tcr` behavior.
- Softmax numerical stability and posterior normalization.
- Abstention threshold behavior (`tau`).
- Missing evidence table behavior.
- Monotonicity checks:
  - `amb_ll_tcr` should increase with stronger mismatch between observed junction UMIs and expected ambient.
  - posterior confidence and abstention behavior should move in expected direction as `tau` changes.

### Small End-to-End Tests (required)
- Run `infer()` on toy AnnData + task config + optional evidence table.
- Assert expected `adata.obs` and `adata.uns` fields.
- Verify one-call automatic ambient routing behavior.
- Validate both paths:
  - `raw_10x_path` provided (raw10x ambient mode)
  - `raw_10x_path` omitted (fallback ambient mode)

### Optional Performance Tests
- Candidate-size scaling checks for inference path.
- Ambient estimator memory/runtime spot checks.

## Tooling and Style
- Formatter: `black`.
- Lint: `ruff`.
- Type checks: `mypy` (module-level target for core package).
- Tests: `pytest`.
- Prefer explicit type hints and dataclasses where they improve readability.
- Avoid sparse-matrix densification in critical paths.

## CI Expectations
Minimum CI for each PR:
- `ruff check .`
- `black --check .`
- `mypy lineageresolver`
- `pytest -q`

CI dataset constraints:
- Tests must run only against committed synthetic fixtures or tiny inline synthetic matrices.
- PRs must not add large binary fixtures or external download dependencies.

## Documentation Discipline
- `docs/SOFTWARE_SPEC.md` is source of truth.
- Keep architecture/config/workflow docs synchronized with behavior changes.
- Ticket acceptance criteria must reflect implemented tests, not aspirational behavior.

## Release Strategy
- Versioning: SemVer (`MAJOR.MINOR.PATCH`).
- `0.x` phase while APIs are still stabilizing.
- Maintain `CHANGELOG.md` with user-facing behavior changes.
- For each release, include:
  - supported commands and API signatures
  - config schema changes
  - migration notes for renamed outputs

## Definition of Release Readiness (v1)
- Required ticket set (`00` to `10`) completed.
- All required tests passing in CI.
- README quickstart and notebook example validated.
- Output schema stable for downstream integration (including future R wrapper path).
